For stacked form fields, prefer [`Field`](/docs/components/field) — it bundles label, description, and error handling. Reach for a standalone `Label` when you pair it directly with a checkbox, switch, or any control where Field's layout is overkill.

## API

### Label

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| `htmlFor` | `string` | — | `id` of the associated control. Clicks on the label focus/activate the control. |
| `required` | `boolean` | `false` | When `true`, renders a red `*` after the label text. |
| `className` | `string` | — | Applied to the root `<label>` |

Built on Radix's `Label.Root` — prevents double-click text selection and forwards clicks to the associated control. All other `<label>` attributes are forwarded.

## Styling

Component tokens:

- `--label-color` — text color (defaults to `--color-text`)
- `--label-font-size` — text size (defaults to `--font-text-sm-size` = 14px)
- `--label-font-weight` — weight (defaults to `--font-weight-medium`)
- `--label-line-height` — line height (defaults to `1`)

Override them at any scope to match a specific surface's typography rhythm.

## When to use Field vs Label

| | Use Field | Use Label |
|---|-----------|-----------|
| Stacked form with label, optional description, and error | ✓ | |
| Single ad-hoc control (checkbox, switch) with a short label | | ✓ |
| Horizontal forms with fixed label width | ✓ | |
| Custom layouts where `Field`'s internal DOM is a constraint | | ✓ |
